\section{Introduction}\label{sec:intro}

Efficient water and environmental control are critical components of modern precision agriculture.  
Traditional irrigation and ventilation scheduling methods in greenhouses are typically manual or rule-based, which can lead to water overuse, plant stress, or inefficient climate regulation.  

In this project we develop a time-aware prediction system that learns, from historical greenhouse sensor data, when:
\begin{itemize}
    \item irrigation should be activated, and
    \item a ventilation fan should be activated,
\end{itemize}
based purely on recent sensor readings and time information.  
The system is trained on a real greenhouse dataset containing temperature, humidity, water level and soil nutrient measurements, together with the actual actuator states (water pump and fan ON/OFF).  

The predictive core of the system is a multi-output binary classifier implemented with a Random Forest inside a scikit-learn pipeline.  
The trained model is exposed through a Flask web application that provides two main services: (i) a ``what to do now'' recommendation given the current sensors and timestamp, and (ii) a 24-hour schedule of expected irrigation and fan activations assuming fixed sensor conditions.  
All experiments and model development are implemented in the Jupyter notebook \texttt{Irrigation\_TimeSeries.ipynb}, while the deployment code lives in \texttt{app/app.py}.

\section{Methodology}\label{sec:methodology}

\subsection{Dataset and Targets}

The dataset consists of chronologically ordered measurements collected from a smart greenhouse between November~2023 and March~2024.  
Each record contains:
\begin{itemize}
    \item a timestamp,
    \item sensor readings: air \textbf{temperature}, \textbf{humidity}, \textbf{water\_level}, and soil nutrients \textbf{N}, \textbf{P}, \textbf{K},
    \item actuator states: irrigation pump ON/OFF and ventilation fan ON/OFF.
\end{itemize}
After basic cleaning (removal of duplicate timestamps and obviously corrupted rows) we construct two binary targets:
\begin{itemize}
    \item $y_{\text{irr}} \in \{0,1\}$: whether irrigation is active,
    \item $y_{\text{fan}} \in \{0,1\}$: whether the ventilation fan is active.
\end{itemize}
These two outputs are predicted simultaneously from the same feature vector.

\subsection{Feature Engineering}

To keep the final model simple and deployable, we restrict the features to quantities that are directly available from sensors or easily derived from the timestamp.  
From each timestamp we extract:
\begin{itemize}
    \item \textbf{month},
    \item \textbf{hour of day},
    \item \textbf{day\_of\_week} (0--6),
    \item a binary \textbf{is\_weekend} flag.
\end{itemize}
From the three nutrient readings we compute a single aggregated index:
\[
\text{nutrient\_index} = \frac{N + P + K}{3}.
\]
The final feature set used by the model is therefore:
\[
\{\text{temperature}, \text{humidity}, \text{water\_level}, \text{month}, \text{hour}, \text{day\_of\_week}, \text{is\_weekend}, \text{nutrient\_index}\}.
\]
These are exactly the features reconstructed from the HTTP request payload in the Flask app before being passed to the model pipeline.

\subsection{Time-Aware Train--Test Split}

Because actuator decisions depend on time and conditions evolve over the season, we avoid random shuffling and respect chronological order.  
The first part of the time series (approximately 80\% of the observations) is used for training, while the final 20\% is kept as a held-out test set that simulates future, unseen data.  
This ensures that evaluation more closely approximates real deployment, where the model must predict future actuator behaviour from past observations.

\subsection{Model Architecture}

The predictive model is implemented as a scikit-learn \texttt{Pipeline} consisting of:
\begin{itemize}
    \item a numeric pre-processing step (imputation and scaling of continuous features), and
    \item a multi-output classifier: \texttt{MultiOutputClassifier} wrapping a \texttt{RandomForestClassifier}.
\end{itemize}
The Random Forest operates directly on the engineered features and produces two probability outputs: one for irrigation activation and one for fan activation.  
Given a feature matrix $X$, the model returns for each sample the estimated probabilities:
\[
P(y_{\text{irr}} = 1 \mid X), \qquad P(y_{\text{fan}} = 1 \mid X).
\]
During deployment these probabilities are compared against a configurable probability threshold (stored in a JSON configuration file) to obtain final ON/OFF decisions.  
The notebook explores several reasonable Random Forest settings (number of trees, depth and regularisation) and settles on a configuration that balances accuracy and overfitting for this dataset.

\subsection{Implementation and Deployment}

Once trained and validated inside \texttt{Irrigation\_TimeSeries.ipynb}, the fitted pipeline is serialised with \texttt{joblib} and saved under \texttt{notebooks/models/model.pkl}, together with a small \texttt{config.json} that records the expected feature order and the decision threshold.  
The Flask application in \texttt{app/app.py} loads this pipeline at start-up and exposes two JSON endpoints:
\begin{itemize}
    \item \texttt{/predict}: computes a ``what to do now'' recommendation from a single timestamp and current sensor readings,
    \item \texttt{/schedule}: given a starting time and constant sensor values, generates a 24-hour schedule of irrigation and fan activations at regular intervals.
\end{itemize}
An HTML front-end (\texttt{templates/index.html}) allows manual input of sensor values and displays the model decisions in a user-friendly way.

\section{Results}\label{sec:results}

We evaluate the model on the held-out, chronologically later portion of the dataset using accuracy, precision, recall and F1-score.  
To study the effect of the decision threshold we test four probability cut-offs (0.80, 0.85, 0.90 and 0.95), and we additionally repeat training with artificial noise added to the input features.

\section{Conclusion}\label{sec:conclusion}

The Smart Agriculture Prediction System demonstrates that a relatively simple, well-engineered multi-output Random Forest can accurately reproduce greenhouse irrigation and ventilation decisions from historical sensor data.  
By respecting temporal structure during training and evaluation, and by carefully aligning the deployed Flask app with the notebook pipeline, the system achieves near-perfect classification performance on a realistic time-based test split and is immediately usable through a lightweight web interface.  

The main strengths of the current approach are its interpretability (through feature importances), robustness to noisy and missing readings thanks to tree-based modelling and preprocessing, and low inference latency suitable for real-time applications.  
Limitations include reliance on a single greenhouse and season, and the absence of explicit long-term temporal dynamics beyond the engineered time features.  
Future work could expand the dataset across more seasons and locations, incorporate additional external data sources such as weather forecasts, and explore more advanced time-series architectures while preserving the practical deployment pathway established in this project.
